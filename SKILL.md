# LLM-Wiki 多知识库管理引擎

## 概述

本 Skill 实现 Karpathy 的 LLM-Wiki 理念——让 LLM 充当知识库管理员，持续维护结构化、互相链接的 Markdown Wiki。知识经过一次"编译"后持续累积，而非每次查询从零检索。

**多知识库架构**：支持创建多个独立的命名知识库，各自维护独立的 raw/wiki/SCHEMA.md 三层结构，互不干扰。通过 `@名称` 语法指定操作目标。

## 多知识库架构

### 目录结构

```
~/WorkBuddy/llm-wiki/                     ← 知识库注册中心（Hub）/ Obsidian Vault 根
├── .obsidian/                  ← Obsidian 配置（全局，所有知识库共享）
├── REGISTRY.md                 ← 知识库注册表
├── AI知识库/                    ← 知识库 "AI知识库"（完全独立）
│   ├── raw/
│   │   └── assets/
│   ├── wiki/
│   │   ├── sources/
│   │   ├── entities/
│   │   ├── concepts/
│   │   ├── comparisons/
│   │   └── analyses/
│   └── SCHEMA.md               ← 该知识库的身份证（Wiki Name + Wiki Root）
├── 职业规划/                    ← 知识库 "职业规划"（完全独立）
│   ├── raw/
│   │   └── assets/
│   ├── wiki/
│   │   ├── sources/
│   │   ├── entities/
│   │   ├── concepts/
│   │   ├── comparisons/
│   │   └── analyses/
│   └── SCHEMA.md
└── ...
```

### `@名称` 语法

所有操作支持 `@名称` 指定目标知识库：

| 用法 | 说明 |
|------|------|
| `#初始化wiki @AI知识库` | 创建名为 "AI知识库" 的知识库 |
| `#摄入 @AI知识库 xxx.pdf` | 向 "AI知识库" 知识库摄入资料 |
| `#查询 @AI知识库 什么是RAG？` | 查询 "AI知识库" 知识库 |
| `#体检 @职业规划` | 对 "职业规划" 知识库做健康检查 |
| `#概览 @AI知识库` | 查看 "AI知识库" 知识库状态 |

### 名称解析规则

1. **有 `@名称`**：直接定位 `~/WorkBuddy/llm-wiki/<名称>/`
2. **无 `@名称`，仅有1个知识库**：自动使用该知识库
3. **无 `@名称`，有多个知识库**：展示友好选择提示（见下方模板），请用户选择
4. **`@名称` 对应的知识库不存在**：提示用户先 `#初始化wiki @名称` 或从列表中选择

#### 多库选择提示模板

当用户未指定 `@名称` 且存在多个知识库时，**必须**输出以下格式的提示：

```
📋 当前有 N 个知识库，请指定操作目标：

| # | 名称 | 页面数 | 最后操作 |
|---|------|--------|----------|
| 1 | AI知识库 | 12 | 2026-04-30 摄入 |
| 2 | 职业规划 | 8 | 2026-04-29 查询 |

💡 用法示例：
  #摄入 @AI知识库 xxx.pdf
  #查询 @职业规划 985学生特点
  #概览 @AI知识库
```

### REGISTRY.md 格式

```markdown
# LLM-Wiki 知识库注册表

| 名称 | 路径 | 创建时间 | 页面数 | 最后操作 |
|------|------|----------|--------|----------|
| AI知识库 | ~/WorkBuddy/llm-wiki/AI知识库/ | 2026-04-28 | 12 | 2026-04-30 摄入 |
| 职业规划 | ~/WorkBuddy/llm-wiki/职业规划/ | 2026-04-28 | 8 | 2026-04-29 查询 |
```

> REGISTRY.md 由 LLM 在每次 Init 时自动更新，不需要用户手动维护。

## 触发词与操作映射

| 触发词 | 操作 | 说明 |
|--------|------|------|
| `#初始化wiki` / `#init` + `@名称` | Init | 创建命名知识库 |
| `#摄入` / `#ingest` + `@名称` + 文件路径或内容 | Ingest | 处理新资料，更新 Wiki |
| `#查询` / `#query` + `@名称` + 问题 | Query | 查询 Wiki 知识 |
| `#体检` / `#lint` + `@名称` | Lint | Wiki 健康检查 |
| `#概览` / `#overview` + `@名称` | Overview | 查看 Wiki 当前状态 |
| `#列表` / `#list` | List | 列出所有知识库 |
| "处理这篇"、"加入知识库"、"整理这篇文章" | Ingest | 自然语言触发 |

| "创建知识库"、"初始化 wiki" | Init | 自然语言触发 |
| "列出知识库"、"有哪些知识库" | List | 自然语言触发 |

## 操作流程

详细规范见 `references/schema.md`。

### List（列出知识库）

**触发条件**：用户想查看有哪些知识库。

1. 读取 `~/WorkBuddy/llm-wiki/REGISTRY.md`（如不存在，说明尚无知识库）
2. 展示注册表中的所有知识库：名称、页面数、最后操作时间
3. 如无知识库，提示用户 `#初始化wiki @名称` 创建第一个

### Init（首次初始化）

**触发条件**：用户首次使用，或要创建新知识库时。

1. 确认知识库名称（从 `@名称` 提取，或询问用户）
2. 确认存放路径（默认：`~/WorkBuddy/llm-wiki/<名称>/`，用户可指定其他路径）
3. 检查 `~/WorkBuddy/llm-wiki/REGISTRY.md` 是否存在——如不存在，先创建 Hub 目录和注册表
4. 检查名称是否已存在——如已存在，提示用户选择其他名称或直接操作已有知识库
5. 创建知识库目录结构：
   ```
   <wiki-root>/
   ├── raw/
   │   └── assets/
   ├── wiki/
   │   ├── sources/
   │   ├── entities/
   │   ├── concepts/
   │   ├── comparisons/
   │   └── analyses/
   └── SCHEMA.md
   ```
6. 将 `references/schema.md` 的内容复制为 `<wiki-root>/SCHEMA.md`，并在文件头部写入：
   ```
   > **Wiki Name**: <名称>
   > **Wiki Root**: <绝对路径>
   ```
7. 初始化 `wiki/index.md`（空索引框架）
8. 初始化 `wiki/log.md`（空日志）
9. 初始化 `wiki/overview.md`（空概览）
10. 更新 `~/WorkBuddy/llm-wiki/REGISTRY.md`，添加新知识库条目
11. **可选 — Git 初始化**：检测系统是否有 `git` 命令，如果有，在知识库根目录执行 `git init` 并做首次提交；如果没有，跳过此步
12. 向用户汇报：知识库创建完成，名称、路径、Git 是否启用，可以开始 `#摄入 @名称`

**Git 检测方法**：执行 `git --version`，成功则说明 Git 可用。

### Ingest（摄入资料）

1. **解析目标知识库**：从 `@名称` 提取，或按"名称解析规则"确定（多库时展示选择提示）
2. 确认资料位置（用户提供文件路径、粘贴内容、或指定 URL）
3. 读取 `<wiki-root>/SCHEMA.md` 获取 Wiki 规范和根目录路径
4. 读取 `wiki/index.md` 了解已有知识结构
5. 读取原始资料内容（如果是 URL，先抓取内容保存到 `raw/`）
6. 生成资料摘要页 → `wiki/sources/YYYY-MM-DD-简短标题.md`
7. 识别实体 → 创建或更新 `wiki/entities/实体名.md`
8. 识别概念 → 创建或更新 `wiki/concepts/概念名.md`
9. 如有对比关系 → 创建 `wiki/comparisons/A-vs-B.md`
10. 更新 `wiki/index.md` 索引
11. 在 `wiki/log.md` 追加操作记录
12. 更新 `REGISTRY.md` 中该知识库的页面数和最后操作
13. **可选 — Git 提交**：如果知识库是 Git 仓库（存在 `.git/` 目录），执行 `git add -A && git commit -m "ingest: 资料标题"`；否则跳过
14. 向用户汇报：处理了什么、新增/更新了哪些页面

### Query（查询知识）

1. **解析目标知识库**：从 `@名称` 提取，或按"名称解析规则"确定（多库时展示选择提示）
2. 读取 `wiki/index.md` 定位相关页面
3. 深入阅读相关页面内容
4. 综合回答，使用 `[[]]` 格式标注引用来源
5. 如答案有长期价值，建议存为新分析页

### Lint（健康检查）

1. **解析目标知识库**：从 `@名称` 提取，或按"名称解析规则"确定（多库时展示选择提示）
2. 遍历 `wiki/` 下所有页面
3. 检查：页面间矛盾、过时信息、孤立页面、缺失引用、frontmatter 完整性
4. 汇报发现的问题和修复建议
5. 用户确认后执行修复
6. 在 `log.md` 记录体检结果
7. 更新 `REGISTRY.md` 中该知识库的最后操作
8. **可选 — Git 提交**：如果知识库是 Git 仓库，执行 `git add -A && git commit -m "lint: 体检结果摘要"`；否则跳过

### Overview（概览）

1. **解析目标知识库**：从 `@名称` 提取，或按"名称解析规则"确定（多库时展示选择提示）
2. 读取 `wiki/index.md` 和 `wiki/log.md`
3. 展示：页面统计、最近操作、知识覆盖情况

## 页面模板

创建 Wiki 页面时，使用 `assets/` 目录下的模板作为起点：

| 模板文件 | 用途 | 何时使用 |
|----------|------|----------|
| `assets/source-template.md` | 资料摘要页 | 每次摄入新资料时 |
| `assets/entity-template.md` | 实体页 | 识别到新实体或更新已有实体时 |
| `assets/concept-template.md` | 概念页 | 识别到新概念或更新已有概念时 |
| `assets/comparison-template.md` | 对比页 | 发现两个实体/概念有对比价值时 |
| `assets/analysis-template.md` | 分析页 | Query 产生有价值的深度分析时 |

模板中的 `{{占位符}}` 在实际创建时替换为真实内容。

## 约束

- **永远不修改 `raw/` 中的文件**
- **每次操作后必须更新 `index.md` 和 `log.md`**
- **Git 是可选组件**：检测到 `.git/` 目录才提交，不强制要求 Git
- **frontmatter 必须完整**（title, type, created, updated, tags, sources）
- 页面间引用使用 Obsidian 双向链接格式 `[[页面路径]]`
- 如发现与已有知识矛盾，在摘要页显式标注
- **Wiki 路径不硬编码**：每次操作从 `SCHEMA.md` 头部读取，或由用户在初始化时指定
- **知识库之间完全独立**：不跨库引用、不跨库搜索，各自维护自己的 raw/wiki/SCHEMA.md
- **REGISTRY.md 是唯一的全局状态**：仅记录名称、路径和统计信息，不存储知识内容
- **多库未指定时必须展示选择提示**：包括编号列表和正确用法示例，引导用户使用 `@名称`
